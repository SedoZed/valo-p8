validExpertises — Documentation technique

Note

Cette documentation s’adresse aux personnes chargées d’installer, configurer, maintenir ou faire évoluer l’application validExpertises. Elle suppose une familiarité de base avec JavaScript, les API REST et Oméka S.

Dépôt GitHub officiel : samszo/validExpertises

Vue d’ensemble

validExpertises est une application web front-end uniquement (HTML + CSS + JavaScript vanilla), conçue pour le Laboratoire Paragraphe de Paris 8. Elle n’a pas de back-end propre : toutes les données sont lues et écrites directement via l’API REST d’Oméka S.

Son rôle dans l’écosystème CartoExpert : permettre à un évaluateur de parcourir des personnes (EC), de consulter leurs mots-clés/concepts d’expertise et d’y attacher une valeur numérique (-100 à +100), stockée sous forme d’item Expertise dans Oméka S.

Version courante : v0.2.3

Architecture du projet

validExpertises/
├── index.html                  ← point d'entrée unique (SPA)
├── CHANGELOG.md
├── GUIDE_OPERATEUR.md
├── FICHE_OPERATEUR_1_PAGE.md
└── assets/
    ├── main.css
    ├── data/
    │   └── expertise.json      ← données statiques de référence
    └── js/
        ├── appUrl.js           ← lecture des paramètres URL
        ├── auth.js             ← gestion de la session et de l'évaluateur
        ├── authParams.js       ← valeurs par défaut de configuration
        ├── omk.js              ← couche d'accès à l'API Oméka S
        └── d3.min.js           ← D3.js (chargé localement)
Mise en garde

L’application est une SPA rendue entièrement côté client. Il n’y a aucun build, aucun bundler (webpack, vite…). Les modifications de fichiers JS/CSS sont effectives immédiatement après rechargement du navigateur.

Prérequis techniques

Élément Description
Serveur web Apache, Nginx, ou python -m http.server (pas de CORS avec file://)
Instance Oméka S Accessible en HTTP(S), avec API REST activée
Clé API Oméka S Avec droits lecture et écriture sur les ressources cibles
Vocabulaires RDF Voir tableau ci-dessous

Vocabulaires et propriétés requis dans Oméka S

Préfixe Propriété Usage dans l’application
skos skos:hasTopConcept Propriété par défaut des mots-clés
dcterms dcterms:subject Propriété complémentaire des mots-clés
dcterms dcterms:creator Lien évaluateur ↔︎ item Expertise
dcterms dcterms:source Lien personne ↔︎ item Expertise
curation curation:rank Stockage de la valeur d’expertise
valo valo:expertise Type d’item Expertise
valo valo:EnseignantChercheur Type de personne autorisé (par défaut)
valo valo:Laboratoire Classe utilisée pour le menu laboratoires
Astuce

Ces préfixes sont configurables depuis l’interface ⚙ Paramètres ou dans authParams.js. L’application n’est pas liée à ces valeurs en dur — elles peuvent être adaptées à d’autres instances Oméka S.

Installation et déploiement

Installation minimale (développement local)

# 1. Cloner le dépôt
git clone https://github.com/samszo/validExpertises.git
cd validExpertises

# 2. Servir le dossier via un serveur web local
python3 -m http.server 8080

# 3. Ouvrir dans le navigateur
# → http://localhost:8080

Déploiement en production

Copier le contenu du dossier validExpertises/ vers l’emplacement servi par votre serveur web. Exemple avec Apache :

cp -r validExpertises/ /var/www/html/outils/validExpertises/
Avertissement

CORS : si Oméka S est sur un domaine différent de celui qui sert l’application, configurer les en-têtes CORS sur Oméka S pour autoriser les requêtes depuis le domaine de l’application.

Configuration

Fichier assets/js/authParams.js

Ce fichier définit les valeurs par défaut de tous les paramètres. Il est chargé à l’initialisation avant la lecture du localStorage.

// Exemple de contenu minimal
const authParams = {
  apiOmk: "https://votre-instance.omeka.fr/api/",
  mail: "admin@univ-paris8.fr",
  ident: "",            // key_identity — NE PAS VERSIONNER
  key: "",              // key_credential — NE PAS VERSIONNER
  creatorId: "",        // ID Oméka S de l'évaluateur par défaut
  personTypeAllow: "valo:EnseignantChercheur",
  personPropExp: "skos:hasTopConcept,dcterms:subject",
  rankProp: "curation:rank"
};
Mise en garde

Sécurité : ne jamais committer key_identity et key_credential dans ce fichier si le dépôt est public. Préférer une injection côté serveur (.htaccess, variable d’environnement, fichier de config non versionné) ou demander à chaque opérateur de saisir ses propres clés via l’interface.

Surcharge via l’interface (localStorage)

Les paramètres saisis via ⚙ Paramètres dans l’UI sont stockés dans le localStorage du navigateur, sous la clé validExpertises_cfg.

// Structure stockée dans localStorage
{
  "apiOmk": "https://...",
  "mail": "...",
  "ident": "...",
  "key": "...",
  "creatorId": "...",
  "personTypeAllow": "...",
  "personPropExp": "...",
  "rankProp": "..."
}

La priorité de résolution des paramètres est : localStorage > authParams.js.

Paramètres URL

L’application lit les paramètres suivants dans l’URL à l’ouverture :

Paramètre Effet
idCreator Surcharge l’ID évaluateur défini dans les paramètres
idAuthor Charge automatiquement la personne Oméka S correspondante

Exemple :

http://localhost:8080/?idCreator=7457&idAuthor=123

Flux API Oméka S

Recherche de personnes (autocomplétion)

GET /items?property[0][property]=dcterms:title
         &property[0][type]=in
         &property[0][text]=<saisie>
         &item_type[]=<personTypeAllow>

Lecture d’une personne

GET /items/{id}

Récupération des mots-clés d’une personne

Pour chaque propriété listée dans personPropExp :

GET /items?property[0][property]=<prop>
         &property[0][type]=res
         &property[0][val]={id_personne}

Recherche des items Expertise existants

GET /items?property[0][property]=dcterms:source
         &property[0][type]=res
         &property[0][val]={id_personne}
         &resource_template_id[]={id_template_expertise}

Création d’un item Expertise

Via omk.createItem(payload) — POST sur /items avec le payload JSON-LD complet incluant le type valo:expertise, les liens dcterms:creator et dcterms:source, et la valeur curation:rank.

Modification d’un item Expertise

Via omk.updateRessource(id, payload) — PUT sur /items/{id}.

Suppression d’un item Expertise

Via omk.deleteItem(id) — DELETE sur /items/{id}.

Points d’attention pour la maintenance

Requêtes synchrones (XMLHttpRequest)

Avertissement

Certaines requêtes de lecture dans omk.js utilisent des XMLHttpRequest synchrones. Ce comportement est déprécié par les navigateurs modernes et peut provoquer des avertissements en console ou des blocages. Une migration vers fetch() avec async/await est recommandée comme évolution prioritaire (voir issue ouverte sur le dépôt).

Mise à jour de D3

D3 est chargé localement depuis assets/js/d3.min.js. En cas de mise à jour de D3, remplacer ce fichier manuellement. Vérifier la compatibilité de l’API D3 utilisée dans le code (la v7 a introduit des breaking changes par rapport aux v5/v6).

Évolution des vocabulaires Oméka S

Si les propriétés ou types Oméka S changent (renommage, suppression), mettre à jour :

  1. authParams.js (valeurs par défaut)
  2. La documentation des vocabulaires requis
  3. Tester les flux de lecture et d’écriture après chaque changement

Gestion des versions

Le fichier CHANGELOG.md à la racine du dépôt tracent les évolutions. Mettre à jour ce fichier à chaque release avec :

  • la version (ex. v0.3.0)
  • la date
  • les ajouts, corrections, changements cassants

Sécurité

Risque Recommandation
Clés API versionnées Ne jamais committer ident/key dans authParams.js si repo public
Clés API trop permissives Créer des clés limitées en lecture/écriture sur les seuls types nécessaires
Accès non contrôlé à l’UI Protéger le dossier par authentification serveur (.htaccess, Nginx auth) si besoin
Exposition via URL Les paramètres idCreator/idAuthor dans l’URL sont lisibles — éviter d’y mettre des infos sensibles

Dépannage technique

L’évaluateur n’apparaît pas dans l’en-tête

Vérifier que creatorId correspond bien à un item Oméka S existant du bon type. Tester directement :

GET /items/{creatorId}

Les laboratoires ne se chargent pas

Vérifier que la classe valo:Laboratoire est présente dans le vocabulaire Oméka S et que des items de ce type existent.

Les items Expertise créés ne sont pas retrouvés

Vérifier que le resource_template_id utilisé pour les items Expertise est cohérent entre la création et la recherche. Si le template a été modifié ou recréé dans Oméka S, son ID aura changé.

Erreurs CORS en console

Access to XMLHttpRequest at 'https://omk...' from origin 'https://...' has been blocked by CORS policy

Configurer les en-têtes CORS sur le serveur Oméka S pour autoriser l’origine de l’application. Sur Apache, ajouter dans la configuration ou le .htaccess d’Oméka S :

Header set Access-Control-Allow-Origin "https://valorisation.humanum-p8.fr"
Header set Access-Control-Allow-Methods "GET, POST, PUT, DELETE, OPTIONS"
Header set Access-Control-Allow-Headers "Content-Type"

Intégration dans l’écosystème CartoExpert

validExpertises est un outil parmi d’autres dans la chaîne de valorisation des données de recherche de Paris 8. Il s’articule notamment avec :

  • cleanKeywords (/outils/validKeyword/) : nettoyage des mots-clés avant évaluation
  • Oméka S : base de données centrale qui stocke toutes les ressources (personnes, concepts, expertises)
  • Laboratoire Paragraphe : producteur et mainteneur de l’outillage

Pour tout ajout ou modification structurelle, se référer à la documentation générale CartoExpert et s’assurer de la cohérence avec les autres outils du même serveur.

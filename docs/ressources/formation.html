Formation Omeka S pour Non-Informaticiens

Guide de prise en main de l’interface d’administration

Auteur·rice

Cédric BOUCLY

Date de publication

13 juillet 2026

Mots clés

Omeka S, base de données, métadonnées, ontologies, formation, initiation


NotePréambule

Cette formation concerne uniquement la gestion de données dans Omeka S. La partie CMS/Site et l’API d’exposition ne sont pas abordées.

Public Cible : Contributeurs non-techniciens devant modifier des données dans Omeka S via l’interface d’administration.

Objectifs : À la fin de la formation, les participants seront capables de :

  1. Comprendre la logique interne d’Omeka S (Items, Item Sets, Relations, etc)
  2. Naviguer dans la base de données sans confusion entre les différents types d’objets
  3. Effectuer 2-3 modifications spécifiques directement dans l’interface Omeka S.

AstuceObjectif

Module 2 — L’Item, Unité de Base

AstuceObjectif

Comprendre qu’un item est l’entrée fondamentale dans la base.

Anatomie d’un Item

Un item = une entrée dans la base de données. Il contient trois parties :

ITEM
Élément Description
ID (o:id) Numéro unique
Métadonnées Champs descriptifs
Médias Fichiers attachés

L’identifiant “o:id”

Définition : Numéro unique attribué automatiquement à la création

Utilité : Ne change jamais, même si le titre ou les métadonnées changent

À retenir : “C’est le numéro de matricule de l’item”

Module 3 — Vocabulaires et Propriétés

AstuceObjectif

Comprendre d’où viennent les champs lorsqu’on ouvre un item.

Principe

Les métadonnées d’un item ne sont pas inventées au hasard. Elles proviennent de vocabulaires, qui définissent :

  • Quels champs existent
  • Ce que chaque champ signifie
  • Quel type de valeur est attendu

Exemple Concret : Dublin Core (dcterms:xxx) → Le vocabulaire le plus utilisé, présent par défaut dans Omeka S, ayant comme propriété…

Propriété Signification Type Attendu
dcterms:title Titre de l’item Texte
dcterms:creator Créateur Texte / Ressource
dcterms:date Date Texte (format ISO recommandé)
dcterms:description Description Texte long
… … …

Deux Types de Valeurs

Certaines propriétés acceptent :

  • Texte libre → On écrit directement (ex: titre, description)
  • Ressource → On pointe vers un autre item (ex: dcterms:creator peut lier à un item “Personne”)
ImportantAttention

Ceci prépare le Module 4 sur les relations.

Où voir cela

Onglet “Vocabularies” dans l’interface. Utile pour comprendre, pas pour modifier quotidiennement.

Module 4 — Organisation et Relations

AstuceObjectif

Comprendre comment les items s’organisent et se lient entre eux.

Partie A : Les Item Sets

Définition

Un Item Set = un regroupement logique d’items (fonds, collection thématique, période, etc.)

Analogie

  • Item → Item Set
  • Un livre → Une étagère
  • Un objet → Une vitrine muséale
  • Une entrée BDD → Une collection logique

Règles Importantes

  • Un item peut appartenir à plusieurs Item Sets
  • Un Item Set ne contient pas de médias directs (seulement des références à des items)

Manipulation - Voir les Item Sets existants - Créer un nouveau Item Set - Ajouter un item à un Item Set

Partie B : Relations Entre Items via o:id

Pourquoi les Relations ? Omeka S n’est pas une simple liste de fiches indépendantes. C’est un réseau de données reliées.

Le Mécanisme Une propriété de type “ressource” permet de pointer vers un autre item plutôt que d’écrire du texte libre.

Exemple

Item “Document historique” (o:id: 142) └── dcterms:creator → Item “Personne” (o:id: 89) La relation est faite via le o:id de l’item cible.

Pourquoi pas juste du Texte ? Relation via o:id Texte Libre Données structurées et interrogables Impossible de faire des requêtes Liens exploitables par l’API Informations perdues Mise à jour centralisée Duplication des données

Démonstration Requise Créer une liaison entre deux items existants via une propriété de type ressource.

AvertissementCas d’Erreur Courant

“J’ai créé une relation mais elle n’apparaît pas sur le site” → Vérifier :

  • La propriété utilisée accepte-t-elle une ressource ?
  • L’item cible est-il publié/visible ?
  • L’API expose-t-elle ce type de relation ?

Module 6 — Pratique : Vos Modifications

AstuceObjectif

Appliquer immédiatement les concepts dans des tâches réelles.

Structure Recommandée Pour chaque modification demandée :

Démonstration par le formateur Reproduction guidée par les stagiaires (sur items de test ou réels)

Validation : vérification que le changement est bien enregistré

Checklist par tâche

  • Identifier l’item concerné (via recherche ou navigation)
  • Ouvrir l’item et repérer la section à modifier
  • Appliquer la modification
  • Sauvegarder
  • Vérifier le résultat dans la liste ou via une recherche

Si une relation est à créer

  • Identifier l’item source et l’item cible
  • Trouver le o:id de l’item cible (dans l’URL ou les infos de l’item)
  • Utiliser la propriété appropriée de type “ressource”
  • Sélectionner l’item cible dans l’interface (ne pas taper manuellement le o:id)
  • Sauvegarder et tester
